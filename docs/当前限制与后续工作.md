# 当前限制与后续工作

## 1. 当前限制

### 1.1 Overlay 仍然是 prompt-augmentation 路线

这意味着：

1. recall 结果会进入 prompt
2. 并非通过 OpenClaw 原生 memory tool trace 暴露

### 1.2 Memory 模式已兼容工具名，但还不是“完美替代”

原因：

1. OpenClaw core 仍有部分 memory 文案和 bootstrap 逻辑写死在 core
2. 当前项目没有改 core

从产品口径看，这一节应理解为：

1. `Replacement` 已经成立，因为底座和工具实现已经切到 OmniMemory
2. 但 `Replacement` 还不是“语义完全纯净”的最终形态
3. 缺的主要是 patch 层，而不是底座替换本身

### 1.3 当前已修复的关键接缝

目前已完成：

1. `memory_search` / `memory_get` tool result 对齐 `jsonResult(..., details)`
2. `before_compaction` 支持通过 `sessionFile` 回读 transcript
3. 默认 `captureRoles` 收敛为只抓 `user`
4. 本地 mock + 真实 OpenClaw Gateway `/tools/invoke` 已通过
5. 真实 Omni API smoke 已通过：
   - `Overlay` 可真实写入并召回
   - `Replacement` 可真实接管 `memory_search / memory_get`

### 1.4 仍然遗留的问题

1. replacement 语义纯净依赖可选 patch；默认不强制 patch
2. 跨进程 dedupe 目前仅做到“持久化 fingerprint”，还没有更强的 cursor-based exactly-once 语义
3. 当前已经是标准本地可安装插件包，但还不是已发布 npm 包
4. 真实 smoke 显示 Omni 的 `run_id` 检索结果仍可能混入其他 session 的低位结果，因此当前还不能把 `Replacement` 宣称为“严格 session 隔离”的生产级替代

说明：

1. `patch:apply` 已能修复：
   - system prompt 的本地 memory 绑定文案
   - replacement 下本地 memory bootstrap 混入
2. 这属于 replacement 增强层，不是第三种用户模式

### 1.5 当前实现默认共享 runtime 在项目根的 `src/` 下

这对本地源码开发非常友好。当前我们已经通过同步脚本把 runtime 内联到了两个插件包根里，所以本地 `openclaw plugins install <path>` 已成立；但若要正式发布 npm 包，仍需继续评估：

1. 是否继续沿用当前“同步产物”模式
2. 是否改成 workspace package / build 产物

---

## 2. 后续工作

### 优先级 A

1. 增强 writeback 去重为更强的 cursor-based 语义
2. 与 Omni 侧确认并修正 `run_id` / session 检索隔离语义
3. 为 patch 脚本增加版本探测和自动兼容报告
4. 把 overlay / replacement 发布为 npm 包

### 优先级 B

1. 增加本地 markdown memory 导入命令
2. 增加 sync-local-memory 工具
3. 增加 richer evidence formatting
4. 让 smoke 脚本对真实本地 Gateway restart 给出更干净的状态报告
5. 推进对 OpenClaw 社区最新插件实践的持续适配，重点参考 memos 插件的生命周期设计

### 优先级 C

1. 推动 OpenClaw core 做通用解耦：
   - active memory plugin prompt awareness
   - bootstrap filter
   - compaction flush delegation
2. 完成对外文档英文化翻译
