# 实现原则

## 1. 产品模式优先于工程细节

对外只保留两种模式：

1. `Overlay`
2. `Replacement`

不对外暴露第三种“Patch 模式”。

Patch 只作为 `Replacement` 的内部增强层。

## 2. 非破坏性优先

默认优先实现 `omnimemory-overlay`。

原因：

1. OpenClaw 仍在快速迭代
2. 用户已有 `MEMORY.md / memory/*.md`
3. 插件应支持并存与回滚

## 3. 双插件而不是单插件 runtime 双模式

当前 OpenClaw 的 `kind: "memory"` 是 manifest 级静态语义，并受 `plugins.slots.memory` 独占选择约束。

因此：

1. `overlay` 与 `memory replacement` 不适合硬塞进一个插件 id
2. 应做成两个插件形态，共享同一套 runtime

## 4. 工具契约优先复用

对于 `omnimemory-memory`，优先兼容：

1. `memory_search`
2. `memory_get`

## 5. Hook 只做它该做的事

### Overlay

1. `before_prompt_build`
2. `agent_end`
3. `before_compaction`
4. `before_reset`

### Replacement

1. memory tools 为主
2. hooks 为辅助

## 6. Replacement 先替换底座，再考虑 patch

当前 `Replacement` 的最小定义是：

1. 接管 `memory_search`
2. 接管 `memory_get`
3. 把记忆底座切换为 OmniMemory

而不是：

1. 先强制 patch OpenClaw core prompt
2. 先强制抑制 bootstrap 本地 memory

这些 patch 可以提升语义纯度，但不是 replacement 的前置条件。

## 7. Fail-open

1. recall 失败不阻断主回答
2. ingest 失败不阻断主回答
3. 插件禁用后 OpenClaw 应恢复原生行为
