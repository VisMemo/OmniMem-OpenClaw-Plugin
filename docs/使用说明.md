# 使用说明

## 1. 项目目标

本项目对外只提供两种产品模式：

1. `Overlay`
2. `Replacement`

内部实现上，它们对应：

1. `Overlay` -> `omnimemory-overlay`
2. `Replacement` -> `omnimemory-memory`

推荐默认从 `Overlay` 开始。

---

## 2. 两种模式的区别

### 2.1 `Overlay`

特点：

1. 不接管 `plugins.slots.memory`
2. 不破坏 OpenClaw 原生 `memory-core`
3. 通过 hooks 做 recall / capture
4. 更适合快速迭代、升级兼容、灰度试点

一句话：

这是“外挂增强版”，不是替换原生 memory。

### 2.2 `Replacement`

特点：

1. `kind: "memory"`
2. 接管 `plugins.slots.memory`
3. 提供 `memory_search / memory_get`
4. 由 agent 决定何时调用记忆工具、查什么 query
5. 记忆底座从本地 markdown memory 切到 OmniMemory

一句话：

这是“底座替换版”。

### 2.3 关于 patch 的定位

`Replacement` 内部可能会额外启用 prompt/bootstrap patch，用来让替换语义更干净。

但这里要注意：

1. patch 不是第三种用户模式
2. patch 只是 `Replacement` 的增强层
3. 默认不把 patch 作为 replacement 的前置要求

原因：

1. OpenClaw 仍在快速迭代
2. system prompt patch 的版本维护成本高
3. 当前即使不 patch，`memory_search / memory_get` 也已经是 OmniMemory 的实现

## 4. 标准安装方式

两个插件根目录现在已经是标准 OpenClaw 本地插件包，可直接安装：

### 4.1 Overlay

```bash
openclaw plugins install /abs/path/to/OmniMem-OpenClaw-Plugin/plugins/omnimemory-overlay
```

### 4.2 Replacement

```bash
openclaw plugins install /abs/path/to/OmniMem-OpenClaw-Plugin/plugins/omnimemory-memory
```

### 4.3 推荐的统一管理入口

如果希望把“安装 / 切换 / 回滚 / smoke”都收敛成一条产品链，推荐直接用项目自带管理脚本：

```bash
cd OmniMem-OpenClaw-Plugin
node scripts/omnimemory-manage.mjs install --mode overlay
node scripts/omnimemory-manage.mjs switch --mode replacement --apply-patch
node scripts/omnimemory-manage.mjs rollback
node scripts/omnimemory-manage.mjs uninstall --mode replacement --revert-patch
```

也可以做一次完整本地 smoke：

```bash
cd OmniMem-OpenClaw-Plugin
node scripts/run-standard-install-smoke.mjs --mode overlay
node scripts/run-standard-install-smoke.mjs --mode replacement
```

## 5. 在 OpenClaw 中启用 Overlay 模式

安装完成后，确保配置中存在：

```json5
{
  plugins: {
    entries: {
      "omnimemory-overlay": {
        enabled: true,
        config: {
          apiKey: "${OMNI_MEMORY_API_KEY}",
          baseUrl: "https://zdfdulpnyaci.sealoshzh.site/api/v1/memory",
          autoRecall: true,
          autoCapture: true,
          captureStrategy: "last_turn"
        }
      }
    }
  }
}
```

### Overlay 模式行为

1. `before_prompt_build`
   - 检索 OmniMemory
   - 将 recall block 注入 prompt
2. `agent_end`
   - 将最近一轮或全 session 写回 OmniMemory
3. `before_compaction`
   - 优先从 `sessionFile` transcript 回读后补写一次
4. `before_reset`
   - reset/new 前尽量保留记忆

---

## 6. 在 OpenClaw 中启用 Replacement 模式

```json5
{
  plugins: {
    slots: {
      memory: "omnimemory-memory"
    },
    entries: {
      "omnimemory-memory": {
        enabled: true,
        config: {
          apiKey: "${OMNI_MEMORY_API_KEY}",
          baseUrl: "https://zdfdulpnyaci.sealoshzh.site/api/v1/memory",
          autoCapture: true,
          captureStrategy: "last_turn"
        }
      }
    }
  }
}
```

### Replacement 模式行为

1. 接管 `plugins.slots.memory`
2. 注册：
   - `memory_search`
   - `memory_get`
3. 追加 system guidance，提示当前 memory provider 为 OmniMemory
4. 可选进行自动写回
5. 当前默认不强制 patch OpenClaw core 的 system prompt / bootstrap 逻辑

这意味着：

1. agent 仍然自己决定何时调用 `memory_search` / `memory_get`
2. 真正执行的底座已经变成 OmniMemory
3. 但 OpenClaw 原生 memory 文案可能仍残留在 system prompt 中

当前还需要额外注意：

1. 真实 smoke 已确认 `Replacement` 的安装、加载、写入、检索链路成立
2. 但 Omni 当前的 `run_id` 检索语义还不能视为严格 session 隔离
3. 因此 `Replacement` 当前更适合作为高级实验替换版，而不是默认推荐给所有用户的稳定模式

### 6.1 Replacement 可选 patch（推荐给高级用户）

为了让 replacement 语义更干净，项目提供手术级 patch 脚本（带备份和回滚）：

```bash
cd OmniMem-OpenClaw-Plugin
npm run patch:status
npm run patch:apply
npm run patch:revert
```

该 patch 只改 OpenClaw core 两个文件：

1. `src/agents/system-prompt.ts`
2. `src/agents/bootstrap-files.ts`

效果：

1. system prompt 不再强绑定 `MEMORY.md + memory/*.md`
2. 当 `plugins.slots.memory = "omnimemory-memory"` 时，自动抑制本地 memory bootstrap 文件注入

可选配置（`omnimemory-memory` 条目下）：

```json5
{
  plugins: {
    entries: {
      "omnimemory-memory": {
        config: {
          suppressLocalMemoryBootstrap: true
        }
      }
    }
  }
}
```

## 7. 配置字段

通用字段：

| 字段 | 含义 |
| --- | --- |
| `apiKey` | OmniMemory API key，支持直接字符串或 `${ENV}` |
| `baseUrl` | OmniMemory API 根路径 |
| `recallScope` | recall 使用的命名空间，支持 `global`、`agent`、`session` |
| `ingestScope` | ingest 使用的命名空间，支持 `global`、`agent`、`session` |
| `sessionScope` | 旧兼容字段；如果设置，会同时作为 recall/ingest 的 scope |
| `searchLimit` | 默认检索数量 |
| `failSilent` | 检索失败是否静默 |
| `timeoutMs` | 请求超时 |
| `autoRecall` | 是否在 hook 中自动召回 |
| `autoCapture` | 是否自动写回 |
| `recallTopK` | recall top-k |
| `recallMinScore` | recall 阈值 |
| `minPromptChars` | prompt 太短时不触发 recall |
| `captureStrategy` | `last_turn` 或 `full_session` |
| `captureRoles` | 允许写回的消息角色 |
| `writeWait` | 写回时是否等待 ingest 完成 |
| `promptBlockTitle` | overlay recall block 标题 |
| `suppressLocalMemoryBootstrap` | replacement patch 启用后，是否抑制本地 memory bootstrap（默认 `true`） |

---

## 8. 推荐默认值

### 对新用户

推荐先用：

1. `omnimemory-overlay`
2. `recallScope = "global"`
3. `ingestScope = "session"`
4. `captureStrategy = "last_turn"`
5. `autoRecall = true`
6. `autoCapture = true`
7. `captureRoles = ["user"]`

也就是：

- recall 在整个 tenant/account 下共享长期记忆
- ingest 仍按当前 session 写入，避免把所有 run 混成一个桶

### 对高级替换用户

推荐：

1. `Replacement`
2. 保持原有本地 memory 文档不删除
3. 用 replacement 模式单独验证后再长期启用

---

## 9. 当前建议

1. 默认不要同时启用两个 Omni 插件
2. 默认不要一开始就替换原生 memory
3. 先 overlay，后 replacement
4. 默认只抓 `user` 写回，避免 assistant self-poisoning
5. 如果走标准产品链，优先用 `scripts/omnimemory-manage.mjs`
5. 不要把 prompt/bootstrap patch 当成第三种模式去理解

## 9. 本地 mock 实验

当前项目已经包含本地实验脚本与真实集成测试。

运行：

```bash
cd OmniMem-OpenClaw-Plugin
npm run doctor
npm test
npm run test:integration
npm run skill:validate
```

`test:integration` 会验证：

1. Overlay 模式下 recall / capture 命中本地 mock Omni server
2. Replacement 模式下 OpenClaw Gateway 真实加载 `omnimemory-memory`
3. `/tools/invoke` 可调用：
   - `memory_search`
   - `memory_get`

## 10. 给 Agent 自动安装

如果目标不是“手工照着改配置”，而是“交给 Agent 自动完成”，优先看：

1. `skills/omnimemory-installer`
2. `docs/Agent托管安装说明.md`

当前这套交付支持：

1. Agent 自动写入 `plugins.load.paths`
2. Agent 自动切换 overlay / replacement
3. Agent 自动做 validate / doctor / restart
4. replacement 下可选自动 apply patch
